# 푸시 알림 전략

import { Callout } from 'nextra/components'

<Callout type="info">
  푸시 알림은 리텐션의 핵심 레버입니다. 과도한 발송은 앱 삭제로 직결되므로, **빈도 제어와 세그먼트 차등**이 무엇보다 중요합니다.
</Callout>

## 핵심 원칙

| 원칙 | 내용 |
|------|------|
| **가치 우선** | 모든 알림은 유저에게 실질적 이득이나 정보를 전달해야 함 |
| **빈도 제한** | 세그먼트별 하루 최대 발송 건수 상한을 엄격히 준수 |
| **DND 존중** | 야간 시간대(22:00~08:00) 마케팅 알림 차단 |
| **딥링크 필수** | 모든 알림은 관련 화면으로 직행하는 딥링크를 포함 |
| **옵트아웃 보장** | 알림 카테고리별 개별 수신 거부 설정 제공 |

## 빈도 제어 정책

### 세그먼트별 일일 최대 발송 건수

| 세그먼트 | 마케팅 알림 | 트랜잭션 알림 | 총 상한 |
|----------|------------|--------------|---------|
| **신규 유저** (가입 7일 이내) | 1건 | 무제한 | 3건 |
| **활성 유저** (7일 내 접속) | 2건 | 무제한 | 5건 |
| **이탈 위험** (8~30일 미접속) | 1건 | 무제한 | 2건 |
| **휴면** (31일 이상 미접속) | 1건 (주 1회) | 무제한 | 1건 |

<Callout type="warning">
  트랜잭션 알림(당첨, 결과 발표, 소원 목표 달성 등)은 유저가 직접 유발한 행동의 결과이므로 빈도 제한에서 제외합니다. 단, 같은 이벤트의 중복 발송은 방지합니다.
</Callout>

### 동일 알림 중복 방지 규칙

- 같은 `gameId`에 대한 "결과 임박" 알림은 1회만 발송
- 공감 배치 알림은 1시간 단위로 묶어서 1건으로 처리
- 당첨 알림은 같은 `prizeId`로 중복 발송 방지

## Do Not Disturb (DND) 정책

| 시간대 | 허용 알림 유형 |
|--------|---------------|
| **22:00 ~ 08:00** | 트랜잭션 알림만 허용 (당첨, 소멸 D-1 등) |
| **08:00 ~ 22:00** | 모든 알림 허용 (빈도 제한 적용) |

- 유저가 직접 DND 시간대를 앱 설정에서 변경 가능
- 기본값: 22:00~08:00

## 세그먼트별 차등 전략

### 신규 유저 (가입 7일 이내)

**목표**: 핵심 루프(게임 참여 → 보상 경험) 최초 완주 유도

| 일차 | 알림 내용 | 딥링크 |
|------|-----------|--------|
| D+0 | 가입 완료 → 웰컴 포인트 지급 안내 | `blockpick://mypage/wallet` |
| D+1 | 오늘의 DAILY 게임 안내 (첫 게임 유도) | `blockpick://game/daily/today` |
| D+3 | 출석 연속 보너스 안내 | `blockpick://attendance` |
| D+7 | 소원 등록 기능 소개 | `blockpick://wish/new` |

### 활성 유저 (7일 내 접속)

**목표**: 습관적 재방문 + 소원/소문 바이럴 참여 유도

- DAILY 게임 오픈 알림 (매일 1회)
- 공감한 소원 게임 개설 알림 (즉시)
- 포인트 소멸 예정 알림 (해당 시)

### 이탈 위험 유저 (8~30일 미접속)

**목표**: 재접속 트리거 제공

- 3일, 7일 미접속 재참여 알림
- 보유 포인트 잔액 강조 메시지
- 새 DAILY 게임 테마 소개 (흥미 유발)

### 휴면 유저 (31일 이상 미접속)

**목표**: 저비용 재활성화

- 주 1회 이하로 제한
- 포인트 소멸 임박 알림 (소멸 예정 시)
- 신규 기능 출시 알림 (PRIME 오픈 등 빅 이벤트)

## 알림 카테고리별 수신 설정

유저가 앱 설정에서 카테고리별로 알림을 개별 제어할 수 있습니다.

| 카테고리 | 기본값 | 설명 |
|----------|--------|------|
| 게임 결과 및 당첨 | ON | 참여한 게임의 결과, 당첨 알림 |
| 소원/소문 활동 | ON | 공감, 게임 개설, 소원 승인 |
| 출석 및 리워드 | ON | 출석체크, 포인트 소멸 |
| 마케팅 | ON | DAILY 오픈, 신규 게임, 이벤트 |
| 재참여 유도 | ON | 미접속 리마인더 |

<Callout type="info">
  **트랜잭션 알림**(당첨, 결과 발표)은 수신 거부 불가입니다. 유저가 참여한 행동의 결과이므로 서비스 운영상 필수입니다.
</Callout>

## 기술 구현 고려사항 (제약 사항)

<Callout type="warning">
  아래 항목은 기술 구현 방향이 아닌, 기획 관점에서 확인이 필요한 전제 조건입니다. 실제 구현 방식은 아키텍트와 협의가 필요합니다.
</Callout>

| 항목 | 기획 요구사항 |
|------|--------------|
| 발송 플랫폼 | FCM(Android) / APNs(iOS) 모두 지원 필요 |
| 개인화 데이터 | 유저명, 상품명, 포인트 잔액을 알림 문구에 동적 삽입 |
| 딥링크 처리 | 앱 미설치 상태에서 수신 시 설치 후 원래 목적지 이동 필요 (Deferred Deep Link) |
| 발송 이력 | 중복 방지를 위한 알림 발송 이력 저장 필요 |
| A/B 테스트 | 문구 효과 측정을 위한 메시지 변형 테스트 필요 |

## 성공 지표 (KPI)

| 지표 | 목표값 | 측정 방법 |
|------|--------|-----------|
| 알림 오픈율 | 15% 이상 | 알림 클릭 수 / 발송 수 |
| 알림 후 게임 참여율 | 30% 이상 | 알림 클릭 후 게임 진입 전환율 |
| 알림 수신 거부율 | 5% 이하 | 월별 옵트아웃 신규 건수 / 전체 발송 대상 |
| 이탈 위험 유저 재활성화율 | 10% 이상 | 재참여 알림 수신 후 7일 내 재접속 비율 |
