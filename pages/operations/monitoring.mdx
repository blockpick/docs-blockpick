# 모니터링/알림

import { Callout } from 'nextra/components'

블록픽 서비스의 서버, 블록체인, 비즈니스 지표를 실시간으로 모니터링하고 이상 징후 발생 시 신속하게 대응하기 위한 전략을 정의합니다.

---

## 서버 모니터링

### AWS CloudWatch 핵심 지표

| 지표 | 경고 임계값 | 심각 임계값 | 설명 |
|------|------------|------------|------|
| **CPU 사용률** | 70% 이상 | 90% 이상 | ECS 태스크 기준 |
| **메모리 사용률** | 75% 이상 | 90% 이상 | ECS 태스크 기준 |
| **API 에러율 (5xx)** | 1% 이상 | 5% 이상 | ALB 기준 |
| **API 응답 시간 (p95)** | 500ms 이상 | 1,000ms 이상 | CloudWatch API Gateway |
| **RDS 연결 수** | 80% 이상 | 95% 이상 | 최대 연결 수 대비 |
| **Redis 메모리** | 70% 이상 | 85% 이상 | ElastiCache 기준 |

### 헬스체크 엔드포인트

| 엔드포인트 | 확인 항목 | 체크 주기 |
|-----------|-----------|----------|
| `GET /health` | 서버 기본 응답 | 30초 |
| `GET /health/db` | PostgreSQL 연결 상태 | 1분 |
| `GET /health/blockchain` | Ronin RPC 연결 상태 | 1분 |
| `GET /health/cache` | Redis 연결 상태 | 1분 |

```json
// /health 응답 예시
{
  "status": "healthy",
  "version": "1.3.0",
  "uptime": 86400,
  "services": {
    "db": "healthy",
    "cache": "healthy",
    "blockchain": "healthy"
  }
}
```

<Callout type="info">
  헬스체크 엔드포인트는 인증 없이 접근 가능하지만, **내부 IP 대역에서만** 호출되도록 ALB 보안 그룹으로 제한합니다.
</Callout>

## 블록체인 헬스체크

Ronin 블록체인 의존성으로 인한 장애를 사전에 감지합니다.

| 항목 | 정상 기준 | 경고 기준 | 조치 |
|------|----------|----------|------|
| **Ronin RPC 응답 시간** | 1초 미만 | 3초 이상 | 예비 RPC 엔드포인트로 자동 전환 |
| **블록 높이 지연** | 최신 블록 기준 5블록 이내 | 10블록 이상 지연 | 게임 결과 처리 일시 중단 + 알림 |
| **트랜잭션 실패율** | 0.5% 미만 | 1% 이상 | 온콜 담당자 즉시 호출 |
| **RPC 연결 가용성** | 99.9% | 99% 미만 | 장애 선언 + 점검 모드 전환 |

```
블록체인 헬스체크 플로우:

1분마다 실행
  │
  ├─ Ronin RPC 핑 → 응답 시간 측정
  │
  ├─ 최신 블록 번호 조회 → 이전 블록과 높이 비교
  │
  ├─ 최근 100개 트랜잭션 실패율 계산
  │
  └─ 이상 감지 시 → CloudWatch 메트릭 기록 + 알림 발송
```

## 비정상 거래 탐지

### 탐지 규칙

| 패턴 | 임계값 | 탐지 방법 | 조치 |
|------|--------|-----------|------|
| **급격한 포인트 증가** | 1시간 내 1,000P 이상 획득 | 실시간 포인트 이벤트 집계 | 계정 임시 동결 + 수동 검토 |
| **비정상 광고 시청** | 분당 3회 이상 광고 보상 요청 | API 호출 빈도 분석 | 광고 보상 지급 차단 + 플래그 |
| **환불 빈도 초과** | 1주일 내 3회 이상 환불 요청 | 결제/환불 이력 조회 | 결제 기능 임시 제한 |
| **동일 IP 다계정** | 동일 IP에서 3개 이상 계정 활동 | IP 기반 집계 | 계정 검토 큐 등록 |
| **당첨 이상 패턴** | 동일 유저 단기간 반복 당첨 | 당첨 이력 통계 분석 | 수동 검토 + 출금 보류 |

<Callout type="warning">
  자동 탐지 규칙은 **오탐(False Positive)** 가능성이 있으므로, 계정 영구 정지는 수동 검토 후에만 진행합니다. 자동 조치는 임시 제한에 그칩니다.
</Callout>

## 알림 채널

| 채널 | 대상 | 용도 | 응답 목표 |
|------|------|------|----------|
| **Slack** (#blockpick-alerts) | 개발/운영팀 전체 | 일반 경고, 배포 알림, 일간 리포트 | 근무 시간 내 30분 |
| **PagerDuty** | 온콜 담당자 | 서비스 장애, 심각 에러, 블록체인 이상 | **5분 이내** (24/7) |
| **이메일** | 팀 전체 + 이해관계자 | 일간/주간 리포트, 월간 요약 | 익일 확인 |
| **SMS** | 온콜 담당자 | PagerDuty 에스컬레이션 보조 | 즉시 |

### 알림 우선순위

| 레벨 | 조건 | 채널 | 예시 |
|------|------|------|------|
| **P1 (Critical)** | 서비스 전면 장애, 데이터 손실 위험 | PagerDuty + SMS | API 전체 다운, DB 연결 불가 |
| **P2 (High)** | 주요 기능 장애, 에러율 5% 초과 | PagerDuty + Slack | 결제 실패율 급증, 게임 생성 불가 |
| **P3 (Medium)** | 성능 저하, 일부 기능 이상 | Slack | API 응답 지연, 캐시 히트율 저하 |
| **P4 (Low)** | 경고 임계값 초과, 비정상 패턴 탐지 | Slack | CPU 70% 초과, 의심 계정 탐지 |

### Grafana — 서버 메트릭

| 패널 | 데이터 소스 | 주요 지표 |
|------|-----------|----------|
| 시스템 개요 | CloudWatch | CPU, 메모리, 네트워크 I/O |
| API 성능 | CloudWatch + X-Ray | 응답 시간, 처리량, 에러율 |
| DB 상태 | CloudWatch RDS | 연결 수, 쿼리 지연, IOPS |
| 블록체인 상태 | 커스텀 메트릭 | RPC 응답 시간, 블록 지연, 트랜잭션 성공률 |
| 캐시 성능 | CloudWatch ElastiCache | 히트율, 메모리 사용률, 명령 처리량 |

### 커스텀 어드민 — 비즈니스 메트릭

| 패널 | 지표 |
|------|------|
| 실시간 현황 | 현재 활성 유저 수, 진행 중인 게임 수, 오늘 참여 횟수 |
| 수익 지표 | 일간/주간/월간 충전 금액, ARPU, 결제 전환율 |

| 유저 행동 | DAU/MAU, 리텐션, 세션 길이, 무료/유료 참여 비율 |

## 인시던트 대응 프로세스

### 대응 단계

```
이상 감지
  │
  ▼
L1: 자동 복구 시도
  ├─ ECS 태스크 자동 재시작
  ├─ ALB 헬스체크 실패 시 트래픽 차단
  └─ 성공 → 해결 / 실패 → L2
  │
  ▼
L2: 온콜 담당자 개입 (목표: 5분 이내)
  ├─ 장애 원인 파악
  ├─ 임시 조치 (점검 모드, 롤백 등)
  └─ 해결 → 사후 분석 / 미해결 → L3
  │
  ▼
L3: 전체 팀 소집
  ├─ CTO 포함 전체 기술팀
  ├─ 주요 이해관계자 상황 공유
  └─ 장애 해결 + 공식 사후 분석 보고서
```

### 장애 커뮤니케이션

| 단계 | 시점 | 내용 | 채널 |
|------|------|------|------|
| **초기 인지** | 장애 발생 후 10분 이내 | 장애 인지 및 조사 중 안내 | Slack + 상태 페이지 |
| **진행 업데이트** | 30분마다 | 진행 상황 및 예상 복구 시간 | Slack + 상태 페이지 |
| **해결 공지** | 복구 직후 | 복구 완료 및 원인 요약 | Slack + 상태 페이지 + 이메일 |
| **사후 분석** | 복구 후 48시간 이내 | 상세 원인 분석 + 재발 방지 계획 | 이메일 + 내부 문서 |

<Callout type="info">
  모든 P1/P2 인시던트는 **사후 분석(Post-mortem)** 보고서를 작성하고 팀 내부에 공유합니다. 비난 없는 문화(blameless postmortem)를 지향합니다.
</Callout>
