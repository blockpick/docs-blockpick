# 에러/오프라인 처리

블록픽 서비스의 에러 시나리오별 UX 처리, 복구 전략, 데이터 정합성 보장 방안입니다.

## 공통 상태 머신

모든 주요 작업(게임 참여, 결제, 블록체인 트랜잭션)은 상태 머신으로 관리합니다.

```
INITIATED → PROCESSING → COMPLETED
                ↓
             FAILED → ROLLED_BACK
```

| 원칙 | 설명 |
|------|------|
| **상태 머신** | 각 작업을 `INITIATED → PROCESSING → COMPLETED / FAILED / ROLLED_BACK`으로 추적 |
| **Idempotency** | 모든 쓰기 API에 멱등성 키(Idempotency Key) 적용, 중복 요청 시 동일 결과 보장 |
| **감사 로그** | 포인트 변동, 결제, 블록체인 트랜잭션 모두 변경 이력(Audit Log) 기록 |
| **Optimistic Lock** | 포인트 잔액 업데이트 시 낙관적 잠금으로 동시성 충돌 방지 |

## 1. 게임 참여 중 네트워크 끊김

게임 참여 과정은 3단계로 나뉘며, 각 단계별 장애 대응이 다릅니다.

### 1-1. 포인트 차감 후 블록체인 기록 전

> 포인트는 차감(deduct)이 아닌 **홀드(hold/예약)** 방식으로 처리합니다.

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│  포인트   │────▶│  Pending  │────▶│ 블록체인  │
│   홀드    │     │   상태    │     │   기록    │
└──────────┘     └──────────┘     └──────────┘
                      │
              네트워크 끊김 발생
                      │
                      ▼
              ┌──────────────┐
              │ 자동 재시도   │
              │ (복구 시)     │
              └──────┬───────┘
                     │ 5분 초과
                     ▼
              ┌──────────────┐
              │ 자동 롤백    │
              │ + 포인트 복구 │
              └──────────────┘
```

**서버 처리:**
- 포인트를 `HOLD` 상태로 예약 (실제 차감 아님)
- `PendingAction` 레코드 생성, 상태 = `PROCESSING`
- 네트워크 복구 시 자동 재시도 (최대 3회, 지수 백오프)
- **5분 내 미완료 시 자동 롤백** → 포인트 `HOLD` 해제, 상태 = `ROLLED_BACK`

**앱 UX:**
- "연결이 불안정합니다. 재연결 중..." 토스트 표시
- 백그라운드에서 재연결 시도 (진행 바 애니메이션)
- 롤백 완료 시 "참여가 취소되었습니다. 포인트가 복구되었습니다." 안내

### 1-2. 블록체인 기록 중

> `PendingTransaction` 모델로 상태를 추적합니다.

```graphql
type PendingTransaction {
  id: ID!
  userId: String!
  txHash: String
  status: TxStatus!  # PENDING | CONFIRMED | FAILED | ROLLED_BACK
  action: String!    # GAME_JOIN, REWARD_CLAIM 등
  payload: JSON!
  createdAt: DateTime!
  updatedAt: DateTime!
  retryCount: Int!
}
```

**복구 플로우:**
1. 앱 재실행 시 `PendingTransaction` 목록 조회
2. 각 트랜잭션의 상태를 서버에 확인
3. `CONFIRMED` → 정상 반영, `FAILED` → 자동 롤백 + 포인트 복구
4. `PENDING` → 서버에서 블록체인 상태 재확인 후 처리

**앱 UX:**
- 앱 시작 시 "미완료 작업을 확인하고 있습니다..." 로딩 표시
- 완료 시 결과에 따라 성공/실패 안내 다이얼로그

### 1-3. 결과 확인 중

참여 자체는 블록체인에 기록 완료된 상태이므로 **결과만 재조회**하면 됩니다.

**처리:**
- 네트워크 복구 시 게임 결과 API 재호출
- 로컬 캐시에 마지막 참여 ID 저장 → 오프라인에서도 "결과 확인 대기" 표시

**앱 UX:**
- "결과를 불러오는 중입니다..." + 새로고침 버튼
- 결과 도착 시 푸시 알림 발송

## 2. 블록체인 장애 (Ronin)

### 2-1. Ronin RPC 무응답

```
요청 → RPC #1 실패 → RPC #2 (Fallback) → RPC #3 (Fallback)
                                                    │
                                              모두 실패 시
                                                    ▼
                                          "블록체인 연결 중" UI
                                          + 백그라운드 재시도
```

| 단계 | 동작 |
|------|------|
| 1차 시도 | Primary RPC URL로 요청 |
| 2차 시도 | Fallback RPC URL #1 (2초 대기 후) |
| 3차 시도 | Fallback RPC URL #2 (4초 대기 후) |
| 모두 실패 | UI에 "블록체인 연결 중" 표시, 30초 간격 백그라운드 재시도 |

**앱 UX:**
- 블록체인 의존 기능 일시 비활성화 (게임 참여 버튼 dim 처리)
- "블록체인 네트워크 연결 중입니다. 잠시 후 다시 시도해 주세요."
- 연결 복구 시 자동으로 기능 활성화

### 2-2. 트랜잭션 장시간 Pending

| 경과 시간 | 동작 |
|-----------|------|
| 0~30초 | "처리 중..." 스피너 표시 |
| 30초~5분 | 30초 간격 폴링, "처리가 지연되고 있습니다" 안내 |
| 5분 초과 | "처리 중입니다. 완료되면 알려드리겠습니다." + **푸시 알림 예약** |
| 10분 초과 | 서버에서 트랜잭션 상태 강제 확인, 필요 시 롤백 |

**앱 UX:**
- 사용자가 앱을 벗어나도 푸시 알림으로 결과 전달
- "내 활동" 탭에서 Pending 트랜잭션 목록 확인 가능

### 2-3. 스마트 컨트랙트 에러

**처리:**
- 컨트랙트 revert 시 즉시 서버에 에러 보고 (에러 코드 + revert reason)
- **포인트 자동 복구** (홀드 해제 또는 차감 취소)
- CS팀에 Slack 알림 자동 발송
- 에러 패턴 집계 → 반복 발생 시 개발팀 긴급 알림

**앱 UX:**
- "처리 중 문제가 발생했습니다. 포인트가 복구되었습니다."
- "문제가 지속되면 고객센터로 문의해 주세요." + 고객센터 바로가기

## 3. 백엔드 서버 장애

### 3-1. GraphQL 타임아웃 / 500 에러

```
요청 실패 → 1초 후 재시도 → 2초 후 재시도 → 4초 후 재시도
                                                    │
                                              3회 모두 실패
                                                    ▼
                                            오프라인 모드 전환
                                          (읽기 전용, 캐시 데이터)
```

**재시도 정책:**
- 최대 3회, 지수 백오프 (1초 → 2초 → 4초)
- 네트워크 에러와 5xx만 재시도, 4xx(클라이언트 에러)는 즉시 반환

**오프라인 모드:**
- 캐시된 게임 목록, 내 활동 내역 표시 (읽기 전용)
- 참여/결제 등 쓰기 작업 버튼 비활성화
- 상단에 "오프라인 모드" 배너 표시
- 백그라운드에서 서버 health check (30초 간격)
- 연결 복구 시 자동으로 정상 모드 전환 + 데이터 갱신

### 3-2. 배포 중 다운타임

**서버 측:**
- 배포 시작 전 health check endpoint에 `maintenance: true` 플래그 설정
- Rolling deployment로 무중단 배포 목표

**앱 UX:**
- `maintenance: true` 감지 시 "점검 중" 전용 화면 표시
- 예상 완료 시간 표시 (서버에서 `estimatedEndTime` 제공)
- 30초 간격으로 health check → 점검 종료 시 자동 복귀

## 4. 광고 리워드 실패

### 4-1. 광고 로드 실패

| 시도 | 동작 |
|------|------|
| 1차 실패 | "광고를 불러올 수 없습니다" + **재시도 버튼** |
| 2차 실패 | "광고를 불러올 수 없습니다. 잠시 후 다시 시도해 주세요." |
| 3차 실패 | "현재 광고를 불러올 수 없습니다. 나중에 다시 시도해 주세요." + 해당 기능 일시 숨김 |

**원인별 분기:**
- AdMob 네트워크 에러 → 재시도 유도
- 광고 인벤토리 없음 (NO_FILL) → "광고 준비 중" 안내, 10분 후 자동 재시도
- 사용자 광고 차단 감지 → "리워드를 받으려면 광고 차단을 해제해 주세요"

### 4-2. 시청 완료 후 검증 실패

> 광고 시청은 완료했지만 서버 검증(SSVA)에서 실패한 경우

**복구 플로우:**
1. AdMob 세션 ID + 사용자 ID로 서버에 **재검증 요청**
2. 서버에서 AdMob Callback 로그 확인
3. 검증 성공 → 리워드 지급
4. 검증 실패 → CS 에스컬레이션 (자동 티켓 생성)

**앱 UX:**
- "리워드 지급을 확인하고 있습니다..." (자동 재검증 중)
- 실패 시 "리워드 지급에 문제가 발생했습니다. 고객센터에서 확인 후 처리해 드리겠습니다."

### 4-3. 세션 만료

- 광고 시청 세션이 만료된 경우 (장시간 대기 등)
- "세션이 만료되었습니다. 다시 시도해 주세요." + 새 광고 로드 자동 시작

## 5. 결제 관련 에러

### 5-1. 결제 완료 → 포인트 미지급

> 인앱결제(IAP) 완료 후 서버 지급이 실패한 경우

```
┌──────────┐     ┌──────────┐     ┌──────────┐
│ 스토어   │────▶│  서버     │────▶│ 포인트   │
│ 결제완료  │     │ 영수증   │     │  지급    │
└──────────┘     │  검증    │     └──────────┘
                 └────┬─────┘
                      │ 실패
                      ▼
              ┌──────────────┐
              │ 미완료 결제   │
              │ 큐에 저장     │
              └──────┬───────┘
                     │ 앱 시작 시
                     ▼
              ┌──────────────┐
              │ 자동 복구    │
              │ (재검증+지급) │
              └──────────────┘
```

**복구 플로우:**
1. 앱 시작 시 스토어의 미완료 구매(unfinished transactions) 확인
2. 서버에 영수증 재검증 요청
3. 검증 성공 → 포인트 지급 + 스토어 트랜잭션 완료 처리
4. 검증 실패 → CS 에스컬레이션

**앱 UX:**
- 앱 시작 시 자동 복구 → "이전 결제가 정상 처리되었습니다. N 포인트가 지급되었습니다."
- 복구 실패 시 "결제 확인 중 문제가 발생했습니다. 고객센터에 문의해 주세요."

### 5-2. 결제 상태 불명확

> 결제 요청 후 응답을 받지 못한 경우 (네트워크 끊김 등)

**처리:**
1. "결제를 확인하고 있습니다..." UI 표시
2. 백그라운드에서 스토어 API로 결제 상태 확인
3. 결제 완료 확인 → 서버 검증 → 포인트 지급
4. 결제 미완료 확인 → "결제가 완료되지 않았습니다." 안내

**앱 UX:**
- 확인 중에는 중복 결제 방지 (결제 버튼 비활성화)
- 결과 확정까지 "확인 중" 상태 유지

## 에러 화면 공통 가이드

### 디자인 원칙

| 원칙 | 설명 |
|------|------|
| **투명성** | 무슨 일이 일어났는지 사용자 언어로 설명 |
| **안심** | 데이터/포인트가 안전하다는 점을 명시 |
| **액션** | 사용자가 할 수 있는 다음 행동을 제시 |
| **자동 복구** | 가능한 한 사용자 개입 없이 자동으로 복구 |

### 에러 화면 구성

```
┌─────────────────────────────┐
│                             │
│         [에러 아이콘]        │
│                             │
│    문제가 발생했습니다.       │
│    포인트는 안전합니다.       │
│                             │
│    ┌─────────────────────┐  │
│    │     다시 시도하기     │  │
│    └─────────────────────┘  │
│                             │
│       고객센터 문의하기      │
│                             │
└─────────────────────────────┘
```

### Retry 정책 요약

| 에러 유형 | 최대 재시도 | 백오프 | 실패 시 동작 |
|-----------|-----------|--------|-------------|
| GraphQL 5xx | 3회 | 지수 (1s, 2s, 4s) | 오프라인 모드 전환 |
| Ronin RPC | 3회 | 지수 (2s, 4s, 8s) | Fallback RPC → "연결 중" UI |
| 블록체인 TX | 폴링 | 30초 간격 | 5분 후 푸시 알림 |
| AdMob 로드 | 3회 | 즉시 (사용자 트리거) | 기능 일시 숨김 |
| 광고 검증 | 1회 (자동) | 즉시 | CS 에스컬레이션 |
| IAP 복구 | 앱 시작 시 | 즉시 | CS 에스컬레이션 |
